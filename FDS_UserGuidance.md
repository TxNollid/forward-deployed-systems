***Forward Deployed Systems -- An Overview***

The premise is around Palantir's "Forward Deployed Engineer" strategy -- airdrop a small, dense, capable team into a client environment where they are very behind and let them cook with new tooling. I want to call this "Forward Deployed Systems" and treat it more of a way of operating and approaching a problem + the toolbox with capable and mature solutions to help

I want to focus on maybe a discovery + SDLC type of engagement and we can build out later. Everything likely looks like OODA loops -- observe, orient, decide, act -- with a structured input / action / output iteration. 

Lets skip discovery bc that I think is both fluid and straight forward: ingest client environment specifics, update our base assumptions, ingest and process data, and create recommendations + business case. Focusing on SDLC, I think one company that gets this right is 8090 and their software factory tooling. It goes from PRD > requirements > architecture > build task building (like jira stories) > iterate through build to product with AI tools. With that I think we can also connect to existing codebases to allow for modification of code as well

I have built these process steps before but have not chained them together yet. Here is how I typically approach this: start with the task and define the clear input, action, output assumptions. We have an input folder with organization of those inputs, usually there is a sub-cleaning step here. This input is used to create a "tracker" of sorts for the task (we should bring a default for how we expect these trackers to be built). I generally like a high level task one, then a lower level of the items being iterated through and then a Q&A section for questions from the agent the human can come back and answer without slowing down the agent. The actions will need instructions and templates to follow. This is highly variable to the task, but take creating the PRD -- we can bring a set of good PRDs, and then instructions to refine that and add in additional details with a iterating loop of AI generating questions and trying to answer them with the human. The Output should be a structured document designed with the specific intent of being consumed as an input(s) into a downstream process, in the case of a PRD, it should be used to define architecture and requirements. I typically like to have a history log of all agent runs with what was done by the agent and any suggestions for improvements. Other items I expect is to have an overview md file that orchestrates the agent to the repo and based on their assigned role/task, guides them to what they need. Additionally, I like to have a skills file for guidance on how to do certain things, like take creating a ppt what colors to use, how to make an excel, how to write code in certain ways. Also I prefer to have a Rules of Engagement type file that gives specific examples of what not to do that shows the task, the wrong assumption made, and the correct one to do instead. Skills and the rules get enhanced by the FDE during the engagement, those can be very personal 

In addition to all that, I do expect some of the toolkit the FDE to bring to include industry processes & roles, industry IT knowledge / preferences including data guidance, custom tool templates, and preferred KPIs. These ideally are a starting point to the initial discovery and can be refined for the specific client. 

I expect the SDLC to follow lots of those loops I iterated earlier. My assumption is that the more detailed we can get that input action output kernel that I just went through, the more we can chain these together. I do expect a more robust question & answer solution that dispatches additional agents but we will get there in time.
